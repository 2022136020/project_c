# ADR-0003: 화면 데이터 갱신 방식 선택

- 상태: Accepted
- 날짜: 2026-05-26 (12주차)
- 결정자: Jooseonghyeon

---

## Context (배경)

구글 서버에 저장된 데이터가 바뀌었을 때, 앱 화면을 자동으로 새로 고치는 방법을 결정해야 한다.
예를 들어 계좌 잔액이 바뀌면 화면에 즉시 반영되어야 한다.
이 방법을 어떻게 구현하느냐에 따라 코드 복잡도가 크게 달라진다.

---

## Decision (결정)

**Flutter 기본 기능만 사용한다.**

구글 서버(Firebase)는 데이터가 바뀌면 앱에 자동으로 알려주는 기능을 제공한다.
이 알림을 받아서 화면을 바로 갱신하는 Flutter 기본 기능을 그대로 활용한다.
별도의 외부 도구를 추가하지 않는다.

---

## Alternatives (대안)

| 대안 | 장점 | 단점 | 제외 이유 |
|------|------|------|-----------|
| Provider | Flutter 공식 권장 도구, 화면 간 데이터 공유 편함 | 추가 학습 필요, 설정 코드 증가 | 현재 화면 수(8개)에서는 추가 도구 도입 효과보다 복잡도가 더 커짐 |
| Riverpod | Provider보다 안전하고 발전된 방식 | 배우는 데 시간이 더 걸림 | 7주 프로젝트에서 굳이 필요하지 않음 |
| BLoC | 데이터 흐름이 명확해 테스트하기 좋음 | 코드 양이 많고 구조가 복잡 | 소규모 앱에 과도한 방식 |

---

## Consequences (결과)

**좋은 점:**
- 외부 도구 추가 없이 구글 서버 데이터를 화면에 바로 연결 가능
- 코드가 단순하고 빠르게 작성 가능

**나쁜 점 / 제약:**
- 화면이 15개 이상으로 늘어나거나 여러 화면이 같은 데이터를 공유해야 할 때는 다른 방식으로 바꿔야 할 수 있음

---

## 60초 요약문

> "저희 팀은 Flutter 기본 기능만 사용해서 화면을 자동으로 갱신합니다.
> Provider나 Riverpod 같은 외부 도구도 고려했지만,
> 지금 화면이 8개 수준이고 구글 서버 데이터를 바로 화면에 연결하면 충분하기 때문에
> 불필요한 도구를 추가하지 않는 것이 더 적합합니다.
> 이 결정은 ADR-0003에 기록되어 있습니다."
