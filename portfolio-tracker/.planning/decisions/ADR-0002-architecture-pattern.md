# ADR-0002: 코드 구조 방식 선택

- 상태: Accepted
- 날짜: 2026-05-19 (11주차)
- 결정자: Jooseonghyeon

---

## Context (배경)

앱 코드를 어떻게 나눠서 관리할지 결정해야 한다.
화면을 보여주는 코드, 계산하는 코드, 데이터를 저장하는 코드를 한 파일에 전부 몰아넣으면
나중에 수정하거나 발표할 때 "이 코드가 왜 여기 있냐"는 질문에 답하기 어려워진다.

---

## Decision (결정)

**코드를 4개 역할로 나눠서 관리한다.**

```
화면     → 사용자가 눈으로 보는 버튼, 목록, 입력창
흐름     → 화면과 데이터를 연결하는 중간 처리
규칙     → 잔액 계산, 수익률 계산 같은 핵심 계산
저장     → 구글 서버에 데이터를 넣고 꺼내는 작업
```

---

## Alternatives (대안)

| 대안 | 장점 | 단점 | 제외 이유 |
|------|------|------|-----------|
| 역할 구분 없이 한 파일에 전부 | 빠르게 작성 가능 | 오류가 생겨도 원인을 찾기 어렵고, 수정할 때 의도치 않게 다른 기능이 망가질 수 있음 | 앱이 커질수록 관리가 불가능해짐 |
| 더 세밀하게 6~8개로 나누기 | 매우 큰 프로젝트에서 유리 | 7주 프로젝트에서 설정하는 데만 시간이 다 걸림 | 규모에 비해 복잡도가 너무 높음 |
| MVC 방식 | 웹 개발에서 익숙한 구조 | Flutter 앱에서는 역할 구분이 불명확해짐 | Flutter 앱 개발 방식과 맞지 않음 |

---

## Consequences (결과)

**좋은 점:**
- 오류가 발생했을 때 화면 문제인지, 계산 문제인지, 저장 문제인지 원인을 빠르게 좁힐 수 있음
- 한 부분을 수정해도 나머지 부분에 영향을 주지 않아 유지보수가 쉬움
- 새 기능을 추가할 때 어느 폴더에 넣어야 할지 기준이 명확해 혼란이 줄어듦
- 코드가 많아져도 전체 구조를 파악하는 시간이 짧아짐

**나쁜 점 / 제약:**
- 처음에 폴더 구조를 잡는 시간이 필요
- 간단한 기능도 파일을 나눠야 해서 파일 수가 늘어남

---

## 60초 요약문

> "저희 앱은 코드를 4개 역할로 나눠서 관리합니다.
> 화면, 흐름, 계산 규칙, 데이터 저장을 각각 다른 폴더에 둡니다.
> 역할을 나누면 오류가 생겼을 때 어느 부분에서 났는지 바로 찾을 수 있고,
> 한 곳을 고쳐도 다른 기능이 망가질 걱정이 줄어들어 유지보수가 쉬워지기 때문입니다.
> 이 결정은 ADR-0002에 기록되어 있습니다."
